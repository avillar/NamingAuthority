Introduction  1 

With the abundance of geospatial resources available today, technologies such as 2 
catalogue services, are necessary to facilitate data discovery, retrieval, and maintenance.  3 
Because of standardized interface specifications, clients of different origins and with 4 
potentially different focuses can access this technology and the geospatial metadata to 5 
which it provides potential access.  Since these interfaces are standardized, a major role 6 
in the development of catalogue1 services is left to developers, defining information 7 
models which can be utilized by these interfaces and yet remain independent of the 8 
underlying metadata.   9 

Several different organizations have provided insight into these kinds of information 10 
models.  They have produced such work as: 11 

• the Geodata Infrastructure North-Rhine Westphalia (GDI NRW) -- which is the basis 12 
for the current Catalogue 2.0.1 Implementation Specification), and  13 

• several profiles of CSW 2.0.1 for ISO19115/ISO19119 and ebRIM 14 

This profile defines an application profile for valid FGDC Content Standard for Digital 15 
Geospatial Metadata (CSDGM) metadata documents as per the FGDC specification2.  16 
This profile, called FGDC CSDGM Catalogue or CSDGM Catalogue, is based on the 17 
requirements of the CSW 2.0.1 specification, and provides extensions to this base in 18 
several areas.  As stated in CSW 2.0.1 (with Corrigendum), a base profile that provides a 19 
basic set of information objects must be supported by each catalogue instance; in 20 
addition, application profiles for different information communities should be specified. 21

1. Scope 3 

This application profile document specifies the interfaces, bindings, and encodings 4 
required to publish and access digital catalogs of metadata for geospatial data that comply 5 
with the given profile. Metadata acts as generalized properties which can be queried and 6 
returned through catalogue services for resource evaluation, and in many cases invocation 7 
or retrieval of the referenced resource.  8 

This document applies only to implementations of CSW supporting the FGDC Content 9 
Standard for Digital Geospatial Metadata (CSDGM) information model and schema.  10 
Implementations of CSW per this application profile are intended for purposes of 11 
publishing, inventory, discovery, and access of dataset and dataset collection resources.  12 
Implementations conforming to this profile are not intended for similar purposes relative 13 
to geospatial services and applications. 14 

2. Conformance 15 

Abstract conformance to the mandatory catalogue service interfaces is described in 16 
Appendix A. Test data and queries are not included in this profile. 17

6.1 Application domain3 1 

This profile defines a catalogue implementation which allows for the discovery, retrieval, 2 
and modification of geospatial metadata.  It is not intended to be all-encompassing or 3 
general purpose in scope.  The catalogue information model used by this profile is based 4 
on FGDC CSDGM specification, and is specifically designed to operate with FGDC-5 
based geospatial metadata. 6 

This application profile has no specific disciplinary focus. All communities working with 7 
these sorts of geospatial information are addressed. Typical communities are surveying, 8 
environment, geology, landscaping, water management, power industry, 9 
telecommunications etc. 10 

The intention is to implement a generally understood information model based on 11 
standard metadata with only a few relationships among the catalogue items. Usage should 12 
be as simple as possible, implementing a set of use cases typical in the geospatial 13 
community. 14 

The requirements of the information model, search properties, details of the results sets 15 
and interfaces were defined in close cooperation with users in various communities as 16 
well as software vendors. 17

An actor is a person, organisation, or external system that plays a role in one or more 1 
interactions with the system. Four actors are identified: 2 

Publisher: A publisher publishes metadata descriptions to a catalogue. By doing so, he 3 
enables the discovery of that description record by a requestor entity. This actor is also 4 
the owner of the geo-resource that he describes. 5 

Broker: This actor is a specialised publisher that publishes and maintains metadata 6 
records on behalf of the owner of georesources. 7 

Requestor: This actor searches for metadata records in a catalogue service, either by 8 
browsing or through more complicated queries. 9 

Catalogue Service: This is a system that handles the discovery and publishing of 10 
metadata entries. Furthermore, this actor has the ability to harvest metadata records from 11 
other catalogue services. 12

The CSW-Transaction Manager Interface defines operations for creating, modifying and 39 
deleting catalogue records. These operations can all performed using a Transaction 40

8.2.3.1 Harvest Operation 44 

The “harvestResource” operation, on the other hand, directs the catalogue to retrieve an 45

The client does not need to be aware of the information model of the catalogue when 48 
using the “harvestResource” operation, since the catalogue itself is doing the work 49 
required to process the information. The client is simply pointing to where the metadata 50

8.2.3.2 Transaction Operation 22 

The “transaction” operation allows a client to formulate a transaction, and send it to the 23 
catalogue to be processed. The transaction may contain metadata records and elements of 24 
the information model that the catalogue understands. To use the transaction operation, 25 
the client must know something about the information model that the catalogue 26

Design rationale 4 

This section of the profile discusses decisions made during the design of a CSW 2.0.1-5 
compliant catalogue server, referred to as “CSW4FGDC”.  6 

1. Information Model 7 

This section details the decisions made regarding support for an information model based 8 
on FGDC metadata content. 9 

1.1 The OpenGIS® Catalogue Implementation Service Specification (2.0.1) states that all 10 
web services must support a common set of operations and operation parameters.  It does 11 
not, however, detail a mandatory set of metadata elements that must be supported.  12 
Several implementations of the specifications have been based around the Dublin Core 13 
(DC) set of elements, but this profile is designed to cover a catalogue service based on 14 
FGDC elements.   15 

Despite the design for FGDC, this profile recommends any implementation of this design 16 
support the inherent FGDC metadata elements as well as the equivalent DC elements as 17 
they pertain to the supported FGDC elements.  To overcome problems in mapping the 18 
elements of the two models to each other in a manner suitable for a catalogue service, this 19 
profile assumes that certain profile-specific elements may be created which serve as an 20 
intermediate mapping between FGDC and DC elements that do not directly map to each 21 
other.  For instance, DC defines a single-value element named “source” while FGDC 22 
defines a set of compound elements named “source”.  The proper mapping might be 23 
dc:source to the “title” element defined within the fgdc:source element.  However, this is 24 
confusing with the fgdc:title element defined in the identification section of the record.  25 
Therefore a new element designated “fgdc:source” is created to map dc:source to the title 26 
element within fgdc:source.  All requests for this element should be made using the 27 
“fgdc” qualification to ensure the proper values are extracted.  Similar tactics should be 28 
used with further information model discrepancies to ensure data integrity and metadata 29 
content support. 30 

2. Harvest Records 31 

This section describes the decisions reached during the design of the harvest records 32 
feature of CSW4FGDC. 33 

2.1 The OpenGIS® Catalogue Implementation Service Specification (2.0.1) details the 34 
optional usage of the verboseResponse attribute of the HarvestRecords operation to allow 35 
for a “brief” or “summary” result set description of each record harvested as a result of 36 
the harvest request.  Because a harvest request can potentially affect several thousand 37 
records, this profile omits the usage of a result set description for harvested records.  The 38 
normal description detailing the total number of harvested records is still required, but the 39 
use of individual record descriptions is not recommended.40

suitable margin of error.  See the section below for more details.  Because of the need for 35 
a primary key, this implementation of the CSW Server assumes that all repository tables 36 
will contain one primary key column and one XML metadata column.  37

C.1.3 Bounding Coordinates 6 

The FGDC element fgdc:bounding contains four sub-elements which represent 7 
bounding coordinates.  Unlike with the sub-elements of fgdc:keywords, these four sub-8 
elements appear only once each in the XML metadata record.  They must still be handled 9 
differently than the rest of the metadata elements of the information model, though.   10 

• If the four coordinates are requested individually, each utilizes their associated 11 
XPath expressions to extract the requested information in the same fashion as the 12 
rest of the elements in the information model.   13 

• If the coordinates are requested through a single fgdc:bounding element request, 14 
the XPath expression associated with this single element is used which contains 15 
the XPath predicate symbol ‘*’ which identifies all sub-elements of the current 16 
element at the next lowest level of the XML Document Object Model (DOM) 17 
tree.   18 

• The resulting data returned from the query execution must be transformed to 19 
match the record schemas supported by this profile. 20

C.2.1.1 CQL Text 28
